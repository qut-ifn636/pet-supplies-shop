# ALB + CI/CD Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a `/api/health` endpoint, restructure GitHub Actions into a two-job test+deploy pipeline targeting two EC2 instances in parallel, provision both instances behind an ALB, and run Artillery load tests that prove traffic distribution.

**Architecture:** Two EC2 instances run the Express backend (via PM2 + Nginx) behind an Application Load Balancer. GitHub Actions uses two jobs: `test` runs once on any `petopia`-labelled runner; `deploy` fans out in parallel to both EC2 runners via a matrix, using `clean: false` so each instance's `.env` (with its unique `INSTANCE_ID`) is never overwritten by the pipeline.

**Tech Stack:** Node.js 22, Express 4, Mocha/Chai (tests), PM2, Nginx, GitHub Actions self-hosted runners, AWS ALB + EC2, Artillery

### Task 2: Rewrite GitHub Actions CI/CD Workflow

**Files:**
- Modify: `.github/workflows/ci.yml`

- [ ] **Step 1: Replace ci.yml with the two-job pipeline**

Overwrite `.github/workflows/ci.yml` entirely with:

```yaml
name: Petopia CI/CD

on:
  push:
    branches:
      - main

jobs:
  test:
    name: Test
    runs-on: [self-hosted, petopia]

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js 22
        uses: actions/setup-node@v4
        with:
          node-version: '22'

      - name: Install backend dependencies
        working-directory: ./backend
        run: yarn install

      - name: Install frontend dependencies
        working-directory: ./frontend
        run: yarn install

      - name: Build frontend
        working-directory: ./frontend
        run: yarn build

      - name: Run backend tests
        working-directory: ./backend
        env:
          MONGO_URI: ${{ secrets.MONGO_URI }}
          JWT_SECRET: ${{ secrets.JWT_SECRET }}
          PORT: ${{ secrets.PORT }}
        run: npm test

  deploy:
    name: Deploy (${{ matrix.instance }})
    needs: test
    strategy:
      matrix:
        instance: [ec2-1, ec2-2]
    runs-on: [self-hosted, "${{ matrix.instance }}"]

    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          clean: false  # Preserve backend/.env so INSTANCE_ID is not erased

      - name: Install backend dependencies
        working-directory: ./backend
        run: yarn install

      - name: Build frontend
        working-directory: ./frontend
        run: |
          yarn install
          yarn build

      - name: Restart backend
        run: pm2 restart petopia-backend
```

- [ ] **Step 2: Commit**

```bash

```

### Task 5: EC2 Instance Setup

> Repeat **all steps** on both EC2 instances via SSH. Change `ec2-1` → `ec2-2` and the runner name on the second instance.

**Prerequisites:**
- Two EC2 instances launched (Amazon Linux 2023 recommended) in the same VPC
- Security group on each EC2 already allows TCP 22 from your IP
- SSH: `ssh -i your-key.pem ec2-user@<EC2_PUBLIC_IP>`

#### 5a. Install runtime dependencies

```bash
# Node.js 22
curl -fsSL https://rpm.nodesource.com/setup_22.x | sudo bash -
sudo dnf install -y nodejs

# Yarn and PM2
sudo npm install -g yarn pm2

# Nginx
sudo dnf install -y nginx

# Verify
node --version    # expect: v22.x.x
pm2 --version
nginx -v
```

#### 5b. Register the GitHub Actions runner

On GitHub: **Repo → Settings → Actions → Runners → New self-hosted runner**
Select Linux / x64. Follow the displayed `mkdir`, `curl`, `tar` commands on the EC2 shell.

When the config script asks for labels, enter (no spaces):

```
petopia,ec2-1
# On EC2-2: petopia,ec2-2
```

When asked for runner name, use `petopia-ec2-1` (or `petopia-ec2-2`).

Install and start the runner as a systemd service:

```bash
sudo ./svc.sh install
sudo ./svc.sh start
sudo ./svc.sh status   # expect: active (running)
```

Confirm on GitHub Settings → Runners: runner appears as **Idle**.

#### 5c. Manually clone into the GitHub Actions workspace

The deploy job checks out code to `~/actions-runner/_work/pet-supplies-shop/pet-supplies-shop/`. Clone there now so PM2 can be configured before the first automated run:

```bash

```

#### 5d. Create `backend/.env` in the workspace

```bash

MONGO_URI=<paste-atlas-connection-string>
```

#### 5e. Build frontend

```bash

```

#### 5f. Start backend with PM2

```bash

pm2 startup    # copy and run the printed sudo command exactly
```

#### 5g. Configure Nginx

```bash
sudo tee /etc/nginx/conf.d/petopia.conf > /dev/null << 'EOF'
server {
    listen 80;

    location / {
        proxy_pass http://localhost:5001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
EOF

sudo nginx -t                       # expect: syntax is ok
sudo systemctl enable --now nginx
sudo systemctl status nginx         # expect: active (running)
```

### Task 6: AWS ALB Setup (AWS Console)

#### 6a. Update EC2 security group

In EC2 → Security Groups, find the EC2 instances' security group:
- **Remove** any inbound rule: TCP 80 from `0.0.0.0/0`
- **Add** inbound rule: TCP 80, source = the ALB security group (you will create it in 6b — come back to update this after 6b)

#### 6b. Create a security group for the ALB

EC2 → Security Groups → Create:
- Name: `petopia-alb-sg`
- Inbound: TCP 80 from `0.0.0.0/0`
- Outbound: All traffic

Now go back to the EC2 security group and set the TCP 80 source to `petopia-alb-sg`.

#### 6c. Create a Target Group

EC2 Console → **Target Groups → Create target group**:
- Target type: **Instances**
- Protocol: **HTTP**, Port: **80**
- Health check path: `/api/health`
- Healthy threshold: **2**, Unhealthy threshold: **2**, Interval: **15 seconds**
- Name: `petopia-targets`
- Click **Next**, select both EC2 instances → **Include as pending below** → **Create target group**

#### 6d. Create the Application Load Balancer

EC2 Console → **Load Balancers → Create Load Balancer → Application Load Balancer**:
- Name: `petopia-alb`
- Scheme: **Internet-facing**, IP type: **IPv4**
- VPC: same VPC as your EC2 instances
- Subnets: select at least 2 availability zones (ALB requirement)
- Security group: `petopia-alb-sg`
- Listener: HTTP:80 → Forward to `petopia-targets`
- Create

#### 6e. Wait for targets to become healthy

EC2 Console → Target Groups → `petopia-targets` → **Targets** tab.

Both instances should show **healthy** within ~1 minute. If unhealthy:
- SSH to the instance: `curl http://localhost/api/health` — must return 200
- `pm2 status` — `petopia-backend` must be online
- `sudo systemctl status nginx` — must be active

#### 6f. Copy the ALB DNS name

EC2 → Load Balancers → `petopia-alb` → **DNS name** field.
Example: `petopia-alb-1234567890.ap-southeast-2.elb.amazonaws.com`

Save this — it is used in Tasks 7, 9, and 10.
